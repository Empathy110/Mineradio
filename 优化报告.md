## 1.1 移除 3 个 Chromium 反节流命令行开关

**优化类型：** CPU 优化（后台放歌时 CPU 占用过高）  
**影响等级：** ⭐⭐⭐⭐⭐ 最高（核心根因修复）

### 问题描述

原代码在 `CHROMIUM_PERFORMANCE_SWITCHES` 数组中向 Chromium 传递了 4 个反节流开关：

```javascript
['disable-background-timer-throttling'],
['disable-renderer-backgrounding'],
['disable-backgrounding-occluded-windows'],
['disable-features', 'CalculateNativeWinOcclusion'],
```

这四个开关的作用分别是：
- `disable-background-timer-throttling`：禁止 Chromium 在后台时对定时器（包括 `requestAnimationFrame`）进行降频
- `disable-renderer-backgrounding`：禁止 Chromium 在后台时暂停渲染进程
- `disable-backgrounding-occluded-windows`：禁止 Chromium 对「被其他窗口完全遮挡」的窗口进行优化
- `disable-features, CalculateNativeWinOcclusion`：禁止 Windows 原生窗口遮挡检测

**这导致了一个严重问题：** 当用户最小化 Mineradio 窗口，或者切换到其他应用时，Chromium 被明确告知「不要降低渲染频率」，因此 3D 粒子渲染循环（Three.js 的 `requestAnimationFrame` → `renderer.render()`）依然以 60fps 以上的速度全速运行。

### 优化方案

将上述 4 个开关从数组中删除。这样做的好处：

1. Chromium 在窗口不可见/最小化时，会自动将 `requestAnimationFrame` 回调频率从 60fps 降至 1-5fps
2. 渲染进程在后台时会被降低优先级
3. `CalculateNativeWinOcclusion` 特性不再被禁用，Windows 原生窗口遮挡检测恢复生效

### 修改后的代码

```javascript
// ★ 性能优化: 移除阻止后台节流的开关，让 Chromium 在后台自动降频
// 保留: autoplay-policy, ignore-gpu-blocklist, enable-gpu-rasterization,
// enable-oop-rasterization, enable-zero-copy, enable-accelerated-2d-canvas,
// force_high_performance_gpu, use-angle, disable-software-rasterizer,
// enable-features=UseSkiaRenderer
const CHROMIUM_PERFORMANCE_SWITCHES = [
  ['autoplay-policy', 'no-user-gesture-required'],
  ['ignore-gpu-blocklist'],
  ['enable-gpu-rasterization'],
  ['enable-oop-rasterization'],
  ['enable-zero-copy'],
  ['enable-accelerated-2d-canvas'],
  ['force_high_performance_gpu'],
  ['use-angle', 'd3d11'],
  ['disable-software-rasterizer'],
  ['enable-features', 'UseSkiaRenderer'],
];
```

### 预估效果

后台放歌时 CPU 占用降低 70-90%（配合已有的 `render-deep-sleep` 机制）。

## 1.2 启用 backgroundThrottling

**优化类型：** CPU 优化  
**影响等级：** ⭐⭐⭐⭐⭐ 最高

### 问题描述

原代码在 `BrowserWindow` 的 `webPreferences` 中设置了 `backgroundThrottling: false`：

```javascript
webPreferences: {
  preload: path.join(__dirname, 'preload.js'),
  contextIsolation: true,
  nodeIntegration: false,
  sandbox: false,
  backgroundThrottling: false,  // ← 明确禁用后台节流
},
```

这个配置项是 Electron 特有的，它控制 Chromium 是否对后台窗口的渲染进程进行节流。设置为 `false` 意味着：即使窗口完全不可见，渲染进程也保持最高优先级运行。

### 优化方案

将 `backgroundThrottling` 改为 `true`：

```javascript
webPreferences: {
  preload: path.join(__dirname, 'preload.js'),
  contextIsolation: true,
  nodeIntegration: false,
  sandbox: false,
  // ★ 性能优化: 启用后台节流。当窗口不可见/最小化时 Chromium 会自动降低渲染频率
  backgroundThrottling: true,
},
```

### 预估效果

与优化 1.1 配合，Chromium 在后台时会将渲染进程降至低优先级，进一步减少 CPU 占用。

## 1.3 新增抽屉式窗口打开动画

**优化类型：** GPU 优化（窗口响应速度 + 动画体验）  
**影响等级：** ⭐⭐⭐ 中

### 功能描述

新增 `showWithDrawerAnimation()` 函数，实现窗口从屏幕底部向上滑入的动画效果。核心实现逻辑：

1. **获取目标位置：** 读取窗口的目标 `bounds`
2. **计算初始位置：** 将窗口放到屏幕底部以下（`screenBottom = 显示器底部 Y 坐标`）
3. **显示窗口并置顶：** `win.show()` + `win.setAlwaysOnTop(true, 'screen-saver')`
4. **动画循环：** 使用 `requestAnimationFrame` 驱动，每帧计算 easeOutCubic 缓动值，逐帧更新窗口 Y 坐标
5. **动画完成：** 取消置顶，发送窗口状态

动画参数：
- 持续时间：220ms
- 缓动函数：easeOutCubic（`1 - (1 - t)^3`）
- 驱动方式：`requestAnimationFrame`

### 新增代码

```javascript
// ★ GPU 优化: 抽屉式窗口打开动画（从屏幕底部向上滑入）
let drawerAnimationActive = false;
let drawerAnimationId = null;

function showWithDrawerAnimation(win) {
  if (!win || win.isDestroyed() || drawerAnimationActive) {
    if (win && !win.isDestroyed()) win.show();
    return;
  }
  drawerAnimationActive = true;
  const targetBounds = win.getBounds();
  const display = screen.getDisplayMatching(targetBounds);
  const screenBottom = display.bounds.y + display.bounds.height;
  const startY = screenBottom;

  win.setBounds({
    x: targetBounds.x,
    y: startY,
    width: targetBounds.width,
    height: targetBounds.height,
  }, false);

  win.show();
  win.setAlwaysOnTop(true, 'screen-saver');

  const duration = 220;
  const startTime = Date.now();

  function animateDrawer() {
    const elapsed = Date.now() - startTime;
    const progress = Math.min(1, elapsed / duration);
    const eased = 1 - Math.pow(1 - progress, 3); // easeOutCubic
    const currentY = Math.round(startY + (targetBounds.y - startY) * eased);

    if (!win.isDestroyed()) {
      win.setBounds({
        x: targetBounds.x,
        y: currentY,
        width: targetBounds.width,
        height: targetBounds.height,
      }, false);
    }

    if (progress < 1) {
      drawerAnimationId = requestAnimationFrame(animateDrawer);
    } else {
      win.setAlwaysOnTop(false);
      drawerAnimationActive = false;
      drawerAnimationId = null;
      sendWindowState(win);
    }
  }

  drawerAnimationId = requestAnimationFrame(animateDrawer);
}

function cancelDrawerAnimation() {
  if (drawerAnimationId) {
    cancelAnimationFrame(drawerAnimationId);
    drawerAnimationId = null;
  }
  drawerAnimationActive = false;
}
```

### 为什么 GPU 开销极小

- 动画仅修改窗口的 `setBounds`（Y 坐标），不涉及 DOM 操作或 CSS 动画
- 窗口移动由操作系统的窗口管理器（DWM）通过 GPU 合成完成
- `requestAnimationFrame` 仅在动画持续的 220ms 内运行，动画结束后自动停止

**影响等级：** ⭐⭐⭐ 中

## 2.2 延迟 `applySystemCertificateAuthorities()` 调用

**优化类型：** 启动速度优化  
**影响等级：** ⭐⭐⭐ 中

### 问题描述

原代码在模块加载时立即调用 `applySystemCertificateAuthorities()`：

```javascript
// 第 124 行，模块加载时立即执行
applySystemCertificateAuthorities();
```

该函数的作用是读取系统 CA 证书并与 Node.js 内置的 TLS 证书链合并。它涉及：
- `tls.getCACertificates('default')` — 读取 Node.js 内置证书
- `tls.getCACertificates('system')` — 读取系统 CA 证书（同步 I/O）
- 合并去重 → `tls.setDefaultCACertificates(merged)`

在模块加载时调用意味着：**每次 `require('server.js')` 都会阻塞等待 CA 证书读取完成**。

### 优化方案

将调用延迟到服务器实际启动（`server.listen()`）之前：

```javascript
// 修改前：模块加载时立即调用
applySystemCertificateAuthorities();

// 修改后：延迟到 listen 前调用
// ★ 性能优化: 延迟到服务器启动前同步加载，不影响 Electron 主进程并行启动
applySystemCertificateAuthorities();

server.listen(PORT, HOST, () => {
  console.log('======================================================');
  console.log(' 粒子音乐可视化 v2  →  http://localhost:' + PORT);
  ...
});
```

### 预估效果

模块加载时节省约 5-15ms 的同步 I/O 时间。在 Electron 主进程启动阶段，`main.js` 通过 `require()` 加载 `server.js`，这个延迟可以减少 Electron 启动关键路径上的阻塞。

## 2.3 延迟加载 `readPackageInfo()` 和 `readUpdateConfig()`

**优化类型：** 启动速度优化  
**影响等级：** ⭐⭐⭐⭐ 高

### 问题描述

原代码在模块顶层（第 63-65 行）同步调用了两个涉及文件 I/O 的函数：

```javascript
const APP_PACKAGE = readPackageInfo();                    // 同步读取 package.json
const APP_VERSION = process.env.MINERADIO_VERSION || APP_PACKAGE.version || '0.9.11';
const UPDATE_CONFIG = readUpdateConfig(APP_PACKAGE);      // 解析更新配置
```

`readPackageInfo()` 内部执行了 `fs.readFileSync(path.join(__dirname, 'package.json'))`，这是一个**同步文件读取**操作，会阻塞事件循环。

根据代码搜索，`APP_VERSION` 在整个文件中被引用 63 次，`UPDATE_CONFIG` 被引用 30 次。虽然它们被广泛使用，但**首次 HTTP 请求到达之前并不需要立即知道版本号和更新配置**。

### 优化方案

将三个顶层常量改为 lazy 初始化模式：

```javascript
// ★ 性能优化: 延迟加载非关键配置，避免同步 I/O 阻塞模块加载
let APP_PACKAGE = null;
let APP_VERSION = null;
let UPDATE_CONFIG = null;

function lazyReadPackageInfo() {
  if (!APP_PACKAGE) {
    APP_PACKAGE = readPackageInfo();
    APP_VERSION = process.env.MINERADIO_VERSION || APP_PACKAGE.version || '0.9.11';
    UPDATE_CONFIG = readUpdateConfig(APP_PACKAGE);
  }
  return APP_PACKAGE;
}

function getAppVersion() {
  lazyReadPackageInfo();
  return APP_VERSION;
}

function getUpdateConfig() {
  lazyReadPackageInfo();
  return UPDATE_CONFIG;
}
```

所有原先引用 `APP_VERSION` 和 `UPDATE_CONFIG` 的地方，现在通过 `getAppVersion()` 和 `getUpdateConfig()` 访问。首次访问时触发一次性初始化，后续访问直接返回缓存值。

### 预估效果

模块加载时节省约 10-30ms 的同步文件 I/O 时间。`require('server.js')` 不再阻塞在 `readFileSync` 上，Electron 可以更快地完成主进程初始化。

## 3.1 移除 `.ctrl-btn` 的 `will-change: transform`

**优化类型：** GPU 优化（减少 GPU 合成层数量）  
**影响等级：** ⭐⭐⭐ 中

### 问题描述

每个控制按钮（播放、暂停、上一首、下一首、音量等）的 CSS 都包含 `will-change: transform`：

```css
/* 修改前 */
.ctrl-btn {
  ...;
  will-change: transform;
}
```

`will-change: transform` 会告诉浏览器：「这个元素即将发生 transform 变化，请提前创建独立的 GPU 合成层」。每个独立 GPU 合成层都需要额外的显存来存储纹理，并且合成器需要额外的计算来合并这些层。

### 优化方案

直接删除 `will-change: transform`：

```css
/* 修改后 */
.ctrl-btn {
  ...;
  /* will-change 已移除 */
}
```

按钮的 `:hover` 和 `:active` 状态变化通过 `transition` 处理，不需要提前创建 GPU 层。

### 预估效果

减少约 10-15 个独立 GPU 合成层（取决于界面上的按钮数量）。

## 3.5 替换 `.modal` 的 `will-change` 为 `translateZ(0)`

**优化类型：** GPU 优化  
**影响等级：** ⭐⭐⭐ 中

### 问题描述

弹窗的 CSS 使用了 `will-change: opacity, transform, filter`：

```css
/* 修改前 */
.modal {
  ...;
  will-change: opacity, transform, filter;
}
```

`will-change: opacity, transform, filter` 会创建包含所有三个属性的 GPU 合成层，开销较大。而弹窗在整个使用过程中只显示很少次数。

### 优化方案

改为 `transform: translateZ(0)`：

```css
/* 修改后 */
.modal {
  ...;
  transform: translateZ(0);
}
```

两者的区别：
- `will-change: opacity, transform, filter`：创建独立的 GPU 合成层，分配显存，即使元素不可见也保持
- `transform: translateZ(0)`：仅提示浏览器使用 GPU 合成，但不强制创建独立层，开销更小

- `contain-intrinsic-size: auto 360px`：为跳过的元素提供一个占位大小，防止滚动条跳动

## 4.1 添加 `visibilitychange` 监听，后台暂停渲染

**优化类型：** CPU/GPU 优化（后台时完全停止渲染）  
**影响等级：** ⭐⭐⭐⭐⭐ 最高

### 问题描述

原代码的渲染循环是一个无限递归的 `requestAnimationFrame` 调用：

```javascript
// 原代码第 132 行和第 139 行
function draw(nowMs) {
  // ... 绘制粒子、封面模糊效果 ...
  requestAnimationFrame(draw);  // ← 在 draw 函数末尾再次调用，永不停止
}

requestAnimationFrame(draw);  // ← 启动循环
```

这个循环从不检查窗口是否可见。即使壁纸窗口被完全遮挡或最小化，`draw()` 函数仍然每帧执行：
- 计算 420-760 个粒子的位置、颜色、透明度
- 创建渐变对象
- 绘制封面模糊效果
- 调用 Canvas 2D API 进行绘制

所有这些操作都在消耗 CPU 和 GPU 资源，而用户完全看不到。

### 优化方案

将循环改为可控模式，通过 `visibilitychange` 事件在窗口不可见时停止：

```javascript
// ★ 性能优化: 窗口不可见时暂停渲染，减少 CPU/GPU 占用
var wallpaperRunning = true;
var wallpaperRafId = null;

function wallpaperLoop(nowMs) {
  if (!wallpaperRunning) return;  // ← 窗口不可见时直接返回，不渲染
  draw(nowMs);
  wallpaperRafId = requestAnimationFrame(wallpaperLoop);
}

document.addEventListener('visibilitychange', function() {
  wallpaperRunning = !document.hidden;
  if (wallpaperRunning && !wallpaperRafId) {
    wallpaperRafId = requestAnimationFrame(wallpaperLoop);  // ← 恢复时重启
  }
});

wallpaperRafId = requestAnimationFrame(wallpaperLoop);
```

同时将 `draw()` 函数末尾的 `requestAnimationFrame(draw)` 调用删除，因为循环控制已上移到 `wallpaperLoop` 中。

### 核心逻辑

1. **进入后台：** `document.hidden` 变为 `true` → `wallpaperRunning = false` → 下一次 `wallpaperLoop` 回调直接 `return`，不调用 `draw()`，也不递归调用 `requestAnimationFrame`
2. **恢复前台：** `document.hidden` 变为 `false` → `wallpaperRunning = true` → 重新调用 `requestAnimationFrame(wallpaperLoop)`，渲染恢复
3. **恢复速度：** `visibilitychange` 事件触发到渲染恢复的延迟 < 1 帧（约 16ms），用户无感知

### 预估效果

壁纸窗口不可见时，Canvas 2D 渲染完全停止，节省大量 CPU 和 GPU 资源。壁纸窗口通常包含 420-760 个粒子需要每帧计算和绘制，停止渲染可节省显著的 CPU 周期。

# 五、现有机制的协同效果

经过代码审查发现，`index.html` 中早就存在一套完整的「深度后台」节能机制，但之前的反节流开关实际上阻止了它生效：

| 机制 | 代码位置 | 作用 | 优化前状态 | 优化后状态 |
|------|----------|------|------------|------------|
| `render-deep-sleep` CSS 类 | `index.html` CSS | 将 canvas-container 设为 `opacity:0` / `visibility:hidden` | 存在，但 rAF 仍高速空转 | rAF 降频 + CSS 隐藏双管齐下 |
| `applyRendererPowerMode()` | `index.html` JS | 窗口失焦/最小化时触发深度睡眠 | 被 Chromium 反节流开关抵消 | Chromium 配合降频，正常生效 |
| `recoverVisualsAfterBackground()` | `index.html` JS | 窗口恢复时毫秒级重启渲染 | 无法体现效果 | 毫秒级恢复，用户体验无感知 |
| `animate()` 主渲染循环 | `index.html` 第 26627 行 | 深度后台时跳过 `renderer.render()` | rAF 回调仍高频执行（60fps 空转） | rAF 降至 1-5fps，跳过渲染 |

**关键洞察：** 之前的代码存在一个矛盾 — JS 层面做了节能优化（`render-deep-sleep`），但 Chromium 层面又通过命令行开关禁用了节能。这就像一边踩刹车一边踩油门。移除反节流开关后，两套机制协同工作，后台放歌时 CPU 占用预计可降低 70-90%。
